# Melodex UX 改造计划

> 目标:让 Melodex 从“功能都在,但要用户自己理解”变成“按音乐任务自然流动”。本计划先处理使用阻力,再处理视觉 polish。

## 用户不用怎么描述

以后遇到 UX 不舒服,不用组织成设计语言。只要给任意一种输入即可:

- 截图圈一块,说“这里烦 / 挤 / 看不懂 / 不该在这 / 容易误触”。
- 说一句真实动作,例如“我想把这张歌单离线,但不知道点哪个”。
- 只给结果感受,例如“我不知道它有没有开始下载”。

我会把它翻译成:任务流、状态反馈、信息层级、控件位置、视觉一致性这五类问题。

## 现状判断

当前主要问题不是缺功能,而是功能语义暴露得太散:

- 同一首歌同时有播放、验活、加歌单、看歌词、缓存到本机、下载到 NAS 等动作,图标很多,用户需要自己猜每个动作的后果。
- “下载到 NAS”和“缓存到本机”都是保存,但生命周期完全不同:一个是服务器长期曲库,一个是当前浏览器/PWA 缓存。现在二者并列,缺少清晰层级。
- 搜索页存在顶部全局搜索和页面内搜索两个入口,跳转和反馈不够统一。
- 歌单页的批量动作已经有了,但状态只体现在按钮文案里,长任务期间用户缺少整体进度感和下一步判断。
- 移动端把“设置”放在主 Tab,而“曲库/离线/歌单”藏在更多里;对音乐 App 来说,入口权重不够自然。
- 视觉上还残留旧的 brutal shadow、边框按钮和 Spotify 风格混用,导致界面性格不统一。

## UX 原则

1. 以任务命名,少让用户理解实现。
   - 用户任务是“找歌、播放、收藏、保存、离线听、管理曲库”,不是“inspect、cache、saveToServer”。

2. 保存动作必须分层。
   - 收藏:加入“我喜欢”,偏好信号。
   - 加歌单:组织歌曲,不等于离线。
   - 下载到 NAS:进入服务器曲库,长期保存。
   - 缓存到本机:当前设备离线播放,受浏览器空间策略影响。

3. 主动作永远明显,次动作收纳。
   - 播放是每行主动作。
   - 保存/更多作为动作组,减少一排图标造成的认知噪音。

4. 长任务必须有可见状态。
   - 搜索验活、整张歌单下载、整张歌单缓存都要显示当前进度、失败数、完成状态。

5. 移动端优先考虑拇指路径。
   - 主 Tab 应服务高频音乐任务:首页、搜索、曲库、离线/我的、更多。设置不应占主入口。

## P0:先解决“用户不知道点了会发生什么”

### 1. 统一保存语义

影响文件:
- `frontend/src/components/SongRow.js`
- `frontend/src/components/MyPlaylist.js`
- `frontend/src/components/OfflineMusic.js`

改造方向:
- 把行内“下载到 NAS”和“缓存到本机”归为同一组“保存”动作,用菜单或紧凑动作组区分目标。
- 已保存状态在歌曲行上用小状态标记表达,例如“NAS”“本机”,而不是只靠按钮变成勾。
- 歌单页保留两个批量入口,但命名和层级固定:
  - 主按钮:播放全部
  - 次按钮:下载到 NAS
  - 次按钮:缓存到本机
  - 危险操作:更多菜单里的删除歌单

验收标准:
- 用户看到一首歌时,能在 3 秒内判断“播放”“保存到哪里”“是否已经保存”。
- “我喜欢”歌单不会出现删除歌单按钮。
- 歌单批量下载/缓存时,能看到当前进度和失败数。

### 2. 搜索页降低技术感

影响文件:
- `frontend/src/components/Download.js`
- `frontend/src/hooks/useLiveCheck.js`

改造方向:
- 将“正在验活并过滤无法播放的结果”改为更贴近用户的状态表达,例如“正在筛掉不可播放的结果”。
- 搜索结果区增加三种明确状态:搜索中、筛选可播放结果中、无可用结果。
- 排序控件默认简化为“推荐排序 / 高音质优先 / 文件更小”,多级排序作为高级能力收起。
- 顶部全局搜索进入下载页后,页面搜索框同步该关键词,避免两个搜索框像两套系统。

验收标准:
- 用户能知道搜索正在做什么,而不是误以为卡住。
- 不懂音频概念的用户也能理解排序选项。
- 同一关键词从顶部搜索和页面搜索触发的结果体验一致。

### 3. 危险动作与失败动作不静默

影响文件:
- `frontend/src/components/MyPlaylist.js`
- `frontend/src/components/OfflineMusic.js`
- `frontend/src/components/SongRow.js`

改造方向:
- 删除、清空、批量失败要有明确反馈。
- 单曲下载失败/缓存失败提供重试,但不要只靠红色图标暗示。
- 后续可引入 toast,避免 `window.alert` 和静默 catch 混用。

验收标准:
- 用户能知道失败发生在哪里,能重试。
- 取消/失败不会让 UI 假装成功。

## P1:重排主要信息架构

### 4. 导航改成音乐任务模型

影响文件:
- `frontend/src/components/Sidebar.js`
- `frontend/src/App.js`

建议导航:
- 发现：首页推荐
- 找歌：搜索下载
- 曲库：NAS 曲库 / 最近播放 / 离线音乐
- 我的：我喜欢 / 自建歌单
- 更多：设置 / 帮助 / 用户管理

移动端主 Tab 建议:
- 首页
- 搜索
- 曲库
- 离线
- 更多

验收标准:
- 音乐相关高频入口不再藏在“更多”后面。
- 设置不占用移动端主 Tab。
- 当前选中项在桌面侧栏和移动抽屉里始终明显。

### 5. 歌单页更像“可操作的音乐集合”

影响文件:
- `frontend/src/components/MyPlaylist.js`
- `frontend/src/components/PlaylistSongs.js`

改造方向:
- 头图区分歌单类型:我喜欢、自建歌单、推荐歌单。
- 推荐歌单和自建歌单的按钮语义统一:播放全部、下载到 NAS、缓存到本机、加入我的歌单。
- 歌单为空时给直接行动入口,例如跳到搜索页,不要只提示“去搜索里点 +”。

验收标准:
- 空歌单能一键去找歌。
- 推荐歌单和自建歌单的主要操作位置一致。

## P2:移动端与 PWA 离线体验

### 6. 离线能力说清楚边界

影响文件:
- `frontend/src/components/OfflineMusic.js`
- `frontend/src/services/offlineAudio.js`

改造方向:
- 离线页明确展示:本机缓存歌曲数、音频体积、站点占用、持久保存状态。
- 当浏览器不支持持久保存或空间不足时,给可理解反馈。
- Android/Chromium 主路径按已验证能力设计;iOS 标注为受限但不阻断主流程。

验收标准:
- 用户知道“本机缓存”不是 NAS 曲库。
- 清空缓存前有明确确认。
- 离线模式只展示可播放缓存,不触发后端请求。

### 7. 移动端播放器避免遮挡和误触

影响文件:
- `frontend/src/contexts/PlayerContext.js`
- `frontend/src/components/Sidebar.js`
- `frontend/src/App.js`

改造方向:
- 迷你播放器、底部 Tab、页面底部 padding 统一计算,避免中间缝隙或遮挡列表末尾。
- 全屏播放器里队列、歌词、封面切换保持单手可达。
- 重要按钮点击区域不小于 44px。

验收标准:
- 手机上列表最后一行不会被播放器或底栏挡住。
- 常用播放控制不需要精准点小图标。

## P3:视觉系统收口

### 8. 去掉混杂风格

影响文件:
- `frontend/src/index.css`
- `frontend/src/components/*.js`

改造方向:
- 去掉旧 `shadow-brutal*` 的视觉残留,统一为 Spotify 暗色、轻边界、hover 高亮。
- 按钮半径、卡片半径、行高、间距统一。
- 状态色只保留少数语义:主色、成功、警告、危险、弱文本。

验收标准:
- 页面不再同时出现“重边框+硬阴影”和“Spotify 暗色卡片”两套语言。
- 桌面和移动端控件密度一致。

## 第一轮建议施工顺序

1. 改导航与移动 Tab 权重。
   - 难度:1-2h
   - 风险:低
   - 收益:用户先找得到入口。

2. 改歌单页头部动作层级和批量状态。
   - 难度:2-4h
   - 风险:中
   - 收益:保存到 NAS / 本机缓存不再混乱。

3. 改 SongRow 行内动作分组。
   - 难度:3-6h
   - 风险:中
   - 收益:搜索结果不再像按钮测试台。

4. 改搜索页状态和排序表达。
   - 难度:2-4h
   - 风险:低
   - 收益:减少“卡住了吗”的误解。

5. 统一视觉系统。
   - 难度:4-8h
   - 风险:中
   - 收益:整体观感从拼装变成一个产品。

## 验证策略

每轮 UX 改动后至少执行:

- `npm.cmd run build`
- `git diff --check`
- 桌面浏览器验证:
  - 首页进入推荐歌单
  - 搜索真实关键词
  - 播放一首歌
  - 加入歌单
  - 歌单批量下载/缓存状态
  - 离线音乐列表与播放
- 移动视口验证:
  - 底部 Tab 与迷你播放器不遮挡
  - 更多抽屉选中态正确
  - 歌曲行动按钮不误触

真机无法覆盖的平台必须标注未验证,尤其是 iOS Safari。
